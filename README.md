Ce projet a été réalisé dans le cadre du TD de Technologies du Web et consistait à développer une API REST en utilisant Node.js avec Express et TypeScript. L’objectif initial était de mettre en place une structure claire comprenant des routes, des contrôleurs et une configuration TypeScript complète. Dans un second temps, il fallait stocker temporairement les données en mémoire, tester l’API avec Postman, puis, pour aller plus loin, ajouter une solution de persistance avec une base de données et mettre en place un CRUD complet.

Pour commencer, j’ai initialisé un projet Node.js et j’y ai ajouté toutes les dépendances nécessaires : Express pour gérer les routes et les requêtes HTTP, TypeScript pour assurer une meilleure structure du code, nodemon et ts-node pour faciliter le développement en rechargement automatique, et dotenv afin de gérer les variables d’environnement comme le port du serveur. J’ai ensuite configuré le fichier tsconfig.json pour définir le dossier source, la sortie du code compilé, ainsi que les options strictes et l’interopérabilité des modules.

Une fois l’environnement prêt, j’ai mis en place la structure du projet avec un répertoire src contenant un fichier principal index.ts, un dossier routes et un dossier controllers. Cette organisation permet de séparer clairement la logique métier des définitions de routes. Dans un premier temps, les utilisateurs étaient simplement stockés dans un tableau en mémoire, ce qui m’a permis de tester rapidement les premières routes avec Postman. Les routes GET /users et POST /users fonctionnaient correctement, ce qui validait la structure de l’API.

Pour améliorer le projet, j’ai ensuite implémenté une véritable persistance avec une base de données SQLite. Ce choix était pertinent car SQLite ne nécessite aucune installation supplémentaire : la base est stockée dans un simple fichier. J’ai créé un fichier database.ts chargé d’initialiser la base et de générer automatiquement une table users avec trois colonnes : un identifiant unique auto-incrémenté, un nom et un email. À partir de ce moment, toute la logique des contrôleurs a été réécrite pour fonctionner avec SQLite. Les requêtes SQL permettent désormais d’ajouter, de lire, de modifier et de supprimer des utilisateurs de manière persistante.

L’API propose maintenant un CRUD complet : on peut ajouter un utilisateur via POST /users, récupérer toute la liste via GET /users, obtenir un utilisateur précis avec GET /users/:id, mettre à jour ses informations grâce à PUT /users/:id et enfin le supprimer avec DELETE /users/:id. Chaque route renvoie une réponse claire en JSON adaptée à la situation (succès, erreur, utilisateur introuvable, email déjà utilisé, etc.).


L’ensemble de l’API a été testé avec Postman afin de vérifier son bon fonctionnement. Les différents tests ont permis de confirmer que les données étaient correctement enregistrées dans la base SQLite et persistaient même après l’arrêt du serveur, contrairement au stockage en mémoire initial. Les routes mises en place répondent toutes comme attendu et respectent le comportement défini dans le cahier des charges du TP.

Ce projet m’a permis d’enrichir mes connaissances sur Node.js, Express et TypeScript, tout en mettant en pratique une architecture backend moderne. J’ai appris à organiser un projet de manière propre, à structurer des routes et des contrôleurs, à gérer des données avec une base SQL et à tester une API. L’ensemble du TP est fonctionnel et conforme aux critères d’évaluation attendues.
